## エグゼクティブサマリー

4名の専門家視点から PRD をレビューした結果、**ビジョンと設計原則は高く評価**される一方、
以下の **3つの構造的リスク** が浮上した。

| # | リスク | 深刻度 | 対処 |
|---|--------|--------|------|
| 1 | **CF 非準拠問題**: dcmodel (gtool-rs) 自身の出力が CF standard_name を持たず、PRD の自動推論戦略が自分の主要ユースケースで機能しない | 致命的 | フォールバック戦略を v0.1 必須要件に |
| 2 | **MVP スコープ過大**: v0.1 の7項目は1人で3-6ヶ月。GUI フレームワーク泥沼リスクあり | 高 | v0.1a/v0.1b に分割 |
| 3 | **欠けている GFD 基本機能**: ホフメラー図、ベクトル場、Equirectangular 投影、エネルギースペクトルが未定義 | 高 | v0.1 に追加 or P1 格上げ |

## 1. 技術スタック評価

### 推奨構成

| レイヤー | 推奨 | 評価 | 理由 |
|---------|------|:----:|------|
| GUI | **egui + egui_dock** | ◎ | ドッキング UI 成熟、wgpu 統合が公式サポート。Iced はドッキング不在で △ |
| 3D | **wgpu** | ◎ | 60fps 余裕。テクスチャ方式（データ→2D tex、カラーマップ→1D LUT）が最適 |
| NetCDF | **netcdf-rs** | ○ | 活発メンテ中 (v0.12.0)。libnetcdf の C 依存はビルド課題 |
| CF 解釈 | **自前実装** | △ | Rust に高レベル CF ライブラリなし。最大のボトルネック |
| スクリプト (MVP) | **Rhai** | ○ | 軽量・安全・配布問題なし |
| スクリプト (将来) | **PyO3** | ○ | 研究者に馴染むが、Python ランタイム配布が課題 |
| コード生成 | **Python テキスト生成** | ◎ | PyO3 不要。文字列として生成→ユーザーが外部 Python で実行 |

### Globe 描画の推奨アーキテクチャ

```
[NetCDF data] → [CPU: 2D array] → [GPU: 2D Texture (lon×lat)]
                                          ↓
[Colormap LUT] → [GPU: 1D Texture] → [Fragment Shader: lookup]
                                          ↓
                                   [Sphere Mesh → Screen]
```

カラーマップ変更 = 1D テクスチャ差し替えのみ。データ更新 = 2D テクスチャ転送のみ。

## 2. UX 設計の課題と改善

### 2.1 自動推論のフォールバック（P0 追加要件）

PRD の「CF standard_name から自動判定」は **dcmodel 出力自体に対して機能しない**。
gtool-rs は standard_name をフィールド変数に付与していない。

**3段階フォールバック戦略を必須化:**

```
Level 1: CF standard_name       → 確信度高い自動描画
Level 2: 変数名ヒューリスティクス → vor* → 渦度, temp*/T → 温度, d* → 偏差(diverging)
         + units + gtool-rs 独自属性 (grid_type, truncation_nm)
Level 3: 次元構造のみ           → 2D=カラーフィル(viridis), 1D=ラインプロット
```

- 推論結果を Inspector に表示: 「"渦度" と推論しました [変更]」
- 学習機能: ユーザーが修正したら同パターンを記憶

### 2.2 球面操作の再設計

**右ドラッグ=断面は非推奨**。Google Earth / Cesium の標準と競合する。

| 操作 | 現 PRD | 改善案 |
|------|--------|--------|
| 左ドラッグ | 回転 | 回転（変更なし） |
| 右ドラッグ | 断面線 | **パン**（標準準拠） |
| スクロール | ズーム | ズーム（変更なし） |
| 断面操作 | ー | **モード切替**: ツールバーのハサミアイコン or `S` キー押下中にドラッグ |
| クリック | 時系列 | **データ依存**: 時間軸あり→時系列、なし→値表示、鉛直軸あり→プロファイル |

**追加要件**: トラックパッド/タッチジェスチャーの仕様を定義すること（macOS ユーザー大半）。

### 2.3 Progressive Disclosure の遷移トリガー

| 遷移 | トリガー | 方式 |
|------|---------|------|
| L0 → L1 | ファイルを開いた瞬間 | 自動 |
| L1 → L2 | Inspector 折りたたみクリック / カラーマップ変更試行 | 手動（ナッジあり） |
| L2 → L3 | Code Panel 展開 / Cmd+K | 手動 |

**ナッジ手法**:
- パネル端に「ティーザー」（Inspector 折りたたみ時に右端に「設定」タブ）
- コンテキスト駆動ヒント（3回同じ操作 → 「コードで自動化できます」）
- **ポップアップ/モーダルは絶対NG**

### 2.4 コード同期の矛盾解決

**GUI マスター方式を推奨:**
- Code Panel は常に GUI 状態のスナップショットを反映（読み取りビュー）
- ユーザーがコードを編集 →「実行」で GUI を上書き
- 編集中は自動更新を一時停止
- 操作のデバウンス: 0.5秒の静止後にコード更新

### 2.5 小画面対応（13インチノートPC）

- Inspector: Viewport にオーバーレイするフローティングパネル
- 画面幅 1280px 以下: Data Browser もオーバーレイに
- ダブルクリックで Viewport 最大化（教員プレゼン時に有用）

## 3. GFD ドメインの欠落要件

### v0.1 に追加すべき（P0 格上げ）

| 要件 | 理由 |
|------|------|
| **Equirectangular 投影** | GrADS デフォルト。佐藤先生ペルソナの移行パスに必須。Map ビューのデフォルトにすべき |
| **ホフメラー図** | spmodel-rs に既存 (`examples/hovmoller.rs`)。dcmodel ワークフロー直結 |
| **ベクトル場（矢印）重ね合わせ** | 風の場の可視化は GFD の基本中の基本 |
| **E(n) エネルギースペクトル** | spmodel-rs に `energy_spectrum()` 実装済み。1D log-log プロットだけ |

### カラーマップ自動選択の修正

| 変数 | PRD 現状 | 修正 |
|------|---------|------|
| 温度（絶対値） | coolwarm | **sequential (inferno 等)** — 温度は常に正 |
| 温度偏差 | ー | **RdBu_r (diverging, 0中心)** |
| `d*` プレフィックス (dvort, dphi) | ー | **自動で diverging** — アノマリ検出ルール |
| 帯状風 u | ー | **RdBu_r** — 東風(負)/西風(正) |

**汎用ルール追加**: `if min*max < 0: diverging, else: sequential`

### スペクトル可視化の前倒し

PRD では「将来」だが、dcmodel がスペクトル法をアイデンティティとする以上、
**E(n) スペクトル図は v0.1、波数フィルタスライダーは v0.2** に前倒すべき。

## 4. プロダクト戦略の課題

### 4.1 MVP スコープの再定義

**v0.1 → v0.1a + v0.1b に分割を推奨:**

```
v0.1a — 技術検証 (1-2ヶ月):
  [ ] wgpu + egui (egui_dock) のスケルトン UI
  [ ] NetCDF D&D → Globe ビュー (回転・ズーム)
  [ ] fludarium のレンダラー移植
  [ ] 最小限のカラーマップ (viridis, RdBu_r)

v0.1b — 使える最小形 (+2-3ヶ月):
  [ ] Data Browser (変数ツリー)
  [ ] Inspector (投影法、カラーマップ、値範囲)
  [ ] タイムスライダー
  [ ] Map ビュー (Equirectangular, Mollweide)
  [ ] 自動推論 (3段階フォールバック)
  [ ] ホフメラー図
  [ ] E(n) スペクトル図 (1D)
```

### 4.2 真の差別化ポイント

Python エコシステム (xarray + hvPlot) の追い上げに対して、
**2-3年で追いつかれない堀** は以下:

1. **dcmodel エコシステムとの深い統合**: スペクトル空間 ↔ 格子点空間の対応表示、
   渦度-発散形式の物理的意味ヒント — Python 汎用ツールには実装されない
2. **「Teach, don't hide」の実装品質**: ドメイン知識の深さで差がつく
3. **単一バイナリ配布**: conda/pip 地獄からの解放。授業で「これをダウンロードして」の一言で済む
4. **起動速度**: ネイティブアプリの即座の起動は授業ライブデモで実感できる差

**注意**: PRD の差別化表で全項目「◎」は楽観的。v0.1 時点では「Zero-config ○」「球面操作 ○」「学習コスト ◎」程度。

### 4.3 fludarium との棲み分け

fludarium に既に球面レンダリング + 再生モードが実装済み。
ユーザーから見ると「同じことをするツールが2つ」は混乱の元。

**明確な棲み分けを PRD に追記すべき:**
- **fludarium**: リアルタイムシミュレーション可視化（パイプ入力、デモ用）
- **GeoScope**: ポスト処理・解析可視化（ファイル入力、研究・教育用）

### 4.4 成功指標の再定義

| 指標 | 測定方法 | 目標 |
|------|---------|------|
| GitHub Stars | GitHub | 6ヶ月で 100+ |
| ダウンロード数 | Releases / Homebrew | 月間 50+ |
| 初回描画時間 | アプリ内計測 | 30秒以内 |
| 平均セッション時間 | opt-in テレメトリ | 5分以上 |
| 電脳倶楽部セミナーでの反応 | アンケート | 「使いたい」70%+ |
| 研究室導入数 | 手動カウント | 初年度 3研究室+ |

「授業準備1/3」「移行意向50%」は測定困難なため、上記に置換。

### 4.5 コミュニティ導入戦略

```
Phase 0 (今): 電脳倶楽部主要メンバーに PRD を見せて早期フィードバック
Phase 1 (v0.1a): spmodel-rs 出力を GeoScope で開くデモ動画
Phase 2 (v0.1b): 電脳倶楽部セミナーでライブデモ
Phase 3 (v0.2):  大学院講義でパイロット導入 (1-2研究室)
```

## 5. リスクマトリクス

| リスク | 確率 | 影響 | 軽減策 |
|--------|:----:|:----:|--------|
| GUI フレームワーク泥沼 (egui の成熟度不足) | 高 | 高 | v0.1a を2ヶ月で判断。ダメなら Tauri (Web フロント) にピボット |
| 「80%完成、残り20%が永遠」問題 | 高 | 高 | fludarium コード再利用を最大化。Inspector/DataBrowser は最小機能で出す |
| CF 非準拠データでの自動推論失敗 | 高 | 中 | 3段階フォールバック + gtool-rs 側の改善を並行 |
| Python エコシステムの追い上げ | 中 | 中 | 補完関係を構築（Python コード生成で xarray に合流） |
| 貢献者不足 (GFD+Rust+GUI の交差点が狭い) | 中 | 中 | 電脳倶楽部の学生を巻き込む。修士サブプロジェクト化 |
| クロスプラットフォームビルド (libnetcdf C依存) | 中 | 低 | macOS + Linux 優先。Windows は後回し |

### 最大の失敗シナリオ

> 「GUI フレームワークの泥沼にハマり、Panoply レベルの完成度に到達しないまま
> モチベーションが枯渇し、放置される」

**対策**: v0.1a (2ヶ月) でピボット判断。「spmodel-rs の出力を Panoply より快適に見られる」
という一点突破で最初のリリースを出す。

## 7. 総合評価

**PRD の完成度: B+ (良好、修正で A に到達可能)**

| 観点 | 評価 | コメント |
|------|:----:|---------|
| ビジョン・原則 | A | 「開いたら見える、触れる、コードになる」は明確で差別化に直結 |
